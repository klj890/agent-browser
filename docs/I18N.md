# I18N & Accessibility (Stage 21)

## What ships

- **Two locales**: `zh` (Simplified Chinese) and `en` (English).
- **Locale resolution** (highest priority first):
  1. `AdminPolicy.uiLocale` (admin pin) — when set to `zh` or `en`, the user toggle becomes read-only and the UI is forced to that language.
  2. User preference — persisted in `<userData>/agent-browser/locale.json` (`{ "user": "auto" | "zh" | "en" }`).
  3. System locale — Electron's `app.getLocale()` mapped via `normalizeSystemLocale()`. Anything starting with `zh` becomes `zh`; everything else falls back to `en`.
- **English is the schema-of-truth** — every key lives in `apps/renderer/src/i18n/messages/en.ts` and any missing entry in `zh.ts` automatically falls back to the English string. Adding a new key in English is enough to ship; translation can land later without breaking the build.
- **No third-party i18n library**. The runtime is a ~100-line `translate(locale, key, vars?)` helper plus a tiny React provider. This matches the project's "no over-engineering" principle for ~80 strings of UI copy.

## A11y additions

- `<html lang>` is updated whenever the effective locale changes — screen readers and spellcheck pick it up automatically.
- Every icon-only button (back/forward/reload, ⚙ settings, 📖 reading-mode, the new-tab `▾`, the tab `×`) has an `aria-label` translated through the same catalog.
- Settings overlay is a `role="dialog"` with `aria-modal="true"` and `aria-label`.
- `@media (prefers-reduced-motion: reduce)` zeroes out all animations and transitions.
- A `:focus-visible` outline on interactive elements ensures keyboard users see the focus ring even when the OS suppresses it.

## Adding a string

1. Add the key/value to `apps/renderer/src/i18n/messages/en.ts`.
2. Use it via the `useT()` hook: `const { t } = useT(); t("settings.foo.bar", { count: 3 })`.
3. (Optional) translate in `apps/renderer/src/i18n/messages/zh.ts`. Skipping this is fine — the runtime falls back to English.
4. Variable interpolation uses `{name}` placeholders. Unknown variables are kept verbatim so missing context surfaces visibly rather than collapsing to an empty string.

## Adding a locale

1. Drop `apps/renderer/src/i18n/messages/<code>.ts` exporting `Partial<Record<MessageKey, string>>`.
2. Register it in the `DICTS` map inside `apps/renderer/src/i18n/translate.ts`.
3. Extend `Locale` and `LOCALE_PREFS` in `apps/main/src/locale.ts` plus the Zod enum in `AdminPolicySchema.uiLocale`.

## Admin pin example

```bash
# Force English for everyone on this machine, regardless of OS or user choice:
agent-browser-admin set uiLocale en
# Unpin (let users choose):
agent-browser-admin set uiLocale auto
```

## Tests

- `apps/main/src/__tests__/locale.test.ts` — resolver priority, persistence roundtrip, corrupt-JSON fallback.
- `apps/renderer/src/__tests__/i18n.test.ts` — interpolation, zh→en fallback, "zh keys are a subset of en" guard.
- `apps/main/src/__tests__/admin-policy.test.ts` — `uiLocale` accepts `auto|zh|en`, rejects others, defaults to `auto`.
