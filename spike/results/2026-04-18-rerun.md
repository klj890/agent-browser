# Stage 2 Spike — re-run 2026-04-18 (keyed allocator)

After the first run failed 3 of 5 targets (`2026-04-18.md`), the registry was
re-implemented to key `@eN` refs by `backendDOMNodeId` (see
`packages/browser-tools/src/ref-registry.ts`) instead of a naive pre-order
counter. Same 5 targets, same 10 snapshots, same CDP path.

## Results

| target       | base_count | consistency |
|--------------|-----------:|------------:|
| github-repo  | 205        | 1.000       |
| static-blog  | 1          | 1.000       |
| mdn-home     | 85         | 1.000       |
| hn           | 227        | 1.000       |
| wikipedia    | 468        | 1.000       |

**All five targets at 1.000 consistency** — exceeds the 0.95 PLAN threshold.

## Interpretation

- Keying by `backendDOMNodeId` is the correct invariant. DOM nodes that persist
  across snapshots keep their ref; new nodes get new refs monotonically; nodes
  that disappear can be GC'd by `sweep(ttlMs)`.
- Counter only advances — a ref number is never re-used within a page lifetime,
  which matches the Agent's mental model: "`@e7` always means the same thing
  until the page navigates."
- `resetLifetime()` is called on `Page.frameNavigated` / reload / tab close.
