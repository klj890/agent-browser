# Stage 2 Spike — ref-stability run 2026-04-18

Tool: `pnpm spike:ref` (playwright-core 1.59.1 + Chromium for Testing 147)
Sampling: 10 snapshots per target at 200ms intervals, pre-order interactive-role walk,
naive `@e{counter}` allocator. Consistency = share of nodes whose
`backendDOMNodeId` appears in run0 AND is assigned the same `@eN` ref on later runs.

## Results

| target       | base_count | consistency |
|--------------|-----------:|------------:|
| github-repo  | 190        | 0.165       |
| static-blog  | 1          | 1.000       |
| mdn-home     | 85         | 0.307       |
| hn           | 227        | 1.000       |
| wikipedia    | 468        | 0.610       |

Threshold (PLAN 附录 C): every target ≥ 0.95. **Failed on 3 of 5 targets.**

## Analysis

- `static-blog` (example.com) and `hn` (pure static markup, no JS-driven updates) pass trivially.
- `github-repo`, `mdn-home`, and `wikipedia` all have JS that mutates the tree between snapshots:
  skeleton loaders, lazy-mounted headers, late-arriving iframes, etc. The pre-order walk then
  sees a slightly different node set each run, so a naive monotonic counter shifts every ref.
- The `backendNodeId` itself is stable across the same page lifetime — the mismatch is purely
  from counter drift.

## Implications for Stage 2.3

The naive counter allocator from PLAN 附录 C is **not** production-viable. Mitigations to
validate in Stage 2.3:

1. **Keyed allocator**: map `backendDOMNodeId -> ref` and reuse the same ref on re-snapshot
   within a session. This makes consistency trivially 1.0 but breaks PLAN's "each snapshot
   resets counter" guarantee; need to decide which invariant matters for the Agent.
2. **Composite key**: `role + name + closest-landmark` as a secondary lookup when `backendNodeId`
   misses (tab reload / navigation).
3. **Fallback via locator**: rely on `act.ts`'s second path (`{role, name, text}`) when ref
   hits a stale entry.

Decision for Stage 2: adopt (1) + (3) — keyed registry inside a page lifetime, semantic
locator fallback across resets. Re-run this spike after the registry change; expect ≥ 0.98
on all targets.
